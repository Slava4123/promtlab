## Critical (блокеры — править до следующего деплоя)

### CR-1 [SECURITY/DATA] — `gorm.Save()` mass-overwrite в `subscription.Downgrade` и `subscription.handleRefund`

- **Локации:**
  - `backend/internal/usecases/subscription/subscription.go:189-191` (`Downgrade` → `s.users.Update(...)`)
  - `backend/internal/usecases/subscription/subscription.go:413-415` (`handleRefund` → `s.users.Update(...)`)
  - `backend/internal/infrastructure/postgres/repository/user_repo.go:81-83` (`Update()` использует `gorm.Save()`)

- **Что:** `s.users.Update(ctx, &models.User{ID: userID, PlanID: "free"})` создаёт **новый** struct с нулевыми полями (`Email=""`, `PasswordHash=""`, `Role=""`, `Status=""`, `Name=""`, `TokenNonce=""`, `referral_code=""` и т.д.), затем `userRepo.Update` вызывает `gorm.Save()`, который при UPDATE выставляет **все** поля. SQL получится:
  ```sql
  UPDATE users SET email='', password_hash='', name='', role='', status='', plan_id='free', ... WHERE id=42
  ```

- **Последствия (real exploit):**
  1. Юзер с активной Pro/Max-подпиской делает refund в кабинете T-Bank.
  2. Webhook `REFUNDED` → `handleRefund` → `subs.GetActiveByUserID` возвращает `ErrNotFound` (подписка уже expired или paused).
  3. Идёт ветка `else { s.users.Update(...) }` (строка 414) → массовое обнуление полей юзера.
  4. **Первый случай:** UPDATE проходит → юзер теряет email, password_hash, role (→ для admin становится regular user). Если он сделает logout → потеряет аккаунт.
  5. **Каждый следующий refund** другого юзера упадёт на `idx_users_email_unique` (UNIQUE constraint, два юзера с `email=''`) → 5xx, T-Bank ретраит, downtime биллинга.
  6. Вариант 2: если есть admin (плановый downgrade Max→Free через `usecases/admin/admin.go`) — admin теряет `role='admin'` тихо.

- **CVSS 3.1:** **9.1** (AV:N / AC:L / PR:N / UI:N / S:U / C:N / I:H / A:H) — webhook без auth, mass data destruction.

- **Best practice:** [GORM v2 — Updates with `Select` для partial UPDATE](https://gorm.io/docs/update.html#Update-Selected-Fields). Цитата: *"By default, `Save` will update all fields, even if they are zero values."* Решение — partial update через `.Updates(map[string]any{...})` или `.Update("col", val)`.

- **Fix:**
  ```go
  // backend/internal/interface/repository/user.go — добавить
  type UserRepository interface {
      // ... существующие методы
      SetPlan(ctx context.Context, userID uint, planID string) error
  }

  // backend/internal/infrastructure/postgres/repository/user_repo.go
  func (r *userRepo) SetPlan(ctx context.Context, userID uint, planID string) error {
      res := r.db.WithContext(ctx).
          Model(&models.User{}).
          Where("id = ?", userID).
          Updates(map[string]any{
              "plan_id":    planID,
              "updated_at": time.Now(),
          })
      if res.Error != nil { return res.Error }
      if res.RowsAffected == 0 { return repo.ErrNotFound }
      return nil
  }

  // backend/internal/usecases/subscription/subscription.go:190 и :414
  if updErr := s.users.SetPlan(ctx, userID, models.PlanFree); updErr != nil {
      slog.ErrorContext(ctx, "subscription.downgrade.set_plan_failed", "user_id", userID, "error", updErr)
      // не фейлим webhook — payment уже succeeded
  }
  ```

- **Effort:** **S** (новый метод + 2 callsites + интеграционный тест на не-обнуление полей).

### CR-2 [RELIABILITY] — 8 background loops + 3 fire-and-forget горутины **без `defer recover()`**

- **Локации:**
  - `backend/internal/usecases/analytics/insights_loop.go:40-53` (`InsightsComputeLoop.run`)
  - `backend/internal/usecases/analytics/cleanup.go:43-56` (`CleanupLoop.run`)
  - `backend/internal/usecases/subscription/renewal.go:87-99` (`RenewalLoop.run`)
  - `backend/internal/usecases/subscription/expiration.go:50-65` (`ExpirationLoop.run`)
  - `backend/internal/usecases/subscription/reminder.go:50-63` (`ReminderLoop.run`)
  - `backend/internal/usecases/trash/purge.go:35-50` (`PurgeLoop.run`)
  - `backend/internal/usecases/engagement/reengagement.go:58-71` (`ReengagementLoop.run`)
  - `backend/internal/usecases/engagement/streak_reminder.go:46-58` (`StreakReminderLoop.run`)
  - `backend/internal/usecases/auth/auth.go:67-73` (`Service.runBackground` — без recover)
  - `backend/internal/usecases/auth/oauth.go:308-314` (`touchLastLogin` — untracked goroutine)
  - `backend/internal/usecases/team/team.go:217-221` (invite email — untracked goroutine)
  - `backend/internal/usecases/share/share.go:238-249, 290-311` (view counter + share_view_log)
  - `backend/internal/usecases/apikey/apikey.go:154-165` (`UpdateLastUsed` — untracked goroutine)

- **Что:** `go l.run()` запускает goroutine, которая в `for { select <-ticker.C: l.compute() }` вызывает code, который может паниковать (nil-deref на User.Plan, panic в template render, panic в `time.AddDate(0, in.Months, 0)` при некорректных данных, send на closed channel). Без `recover()` panic убивает горутину loop'а, и **loop тихо умирает на весь lifetime процесса**.

- **Последствия:** silent multi-day downtime разных систем, никем не замеченный до alarm'а от пользователей:
  - **RenewalLoop** — подписки не продлеваются → потеря revenue.
  - **ExpirationLoop** — истекшие подписки не помечаются → юзеры на Max после истечения оплаченного периода.
  - **PurgeLoop** — корзина не очищается → рост БД.
  - **ReminderLoop** / **ReengagementLoop** / **StreakReminderLoop** — emails не отправляются → потеря retention.
  - **InsightsComputeLoop** — Max-юзеры не получают обновления Smart Insights.
  - **CleanupLoop** — retention-cleanup не идёт → раздутие `prompt_usage_log`, `share_view_log`.

  Дополнительно: `runBackground` в `auth.Service` при панике (например, `s.email = nil` race) кладёт **весь сервер** (нет `defer recover` → panic propagates до runtime). Docker-compose рестартует за ~10s, но все in-flight HTTP запросы пропадают.

- **Best practice:** [Uber Go Style Guide — Goroutines](https://github.com/uber-go/guide/blob/master/style.md#dont-fire-and-forget-goroutines) + [Dave Cheney — Practical Go](https://dave.cheney.net/practical-go/presentations/qcon-china.html). Цитата: *"Every background goroutine should defer a panic handler."* Также [k8s controller-runtime](https://github.com/kubernetes-sigs/controller-runtime/blob/main/pkg/internal/controller/controller.go#L312) — `runtime.HandleCrash()`.

- **Fix:** Создать общий helper:
  ```go
  // backend/internal/pkg/safeloop/safeloop.go
  package safeloop

  import (
      "log/slog"
      "runtime/debug"

      "github.com/prometheus/client_golang/prometheus"
  )

  var loopPanics = prometheus.NewCounterVec(prometheus.CounterOpts{
      Name: "promptvault_loop_panics_total",
      Help: "Background loop panics that were recovered",
  }, []string{"loop"})

  func init() { prometheus.MustRegister(loopPanics) }

  func RunWithRecover(name string, fn func()) {
      defer func() {
          if r := recover(); r != nil {
              slog.Error("loop.panic",
                  "loop", name,
                  "recover", r,
                  "stack", string(debug.Stack()))
              loopPanics.WithLabelValues(name).Inc()
          }
      }()
      fn()
  }
  ```

  В каждом loop:
  ```go
  // analytics/insights_loop.go:48
  for {
      select {
      case <-ticker.C:
          safeloop.RunWithRecover("insights_loop", l.compute)
      case <-l.stopCh:
          return
      }
  }
  ```

  Для `runBackground`/`touchLastLogin`/`apikey.UpdateLastUsed`/`team.invite-email` ввести `pkg/background.Pool` (см. CR-7) либо аналог recover-обёртки.

- **Effort:** **M** (8 loops × 5 минут + 1 общий helper + worker-pool для fire-and-forget = ~1.5 часа).

### CR-3 [SECURITY] — Destructive admin actions без TOTP: `FreezeUser`, `UnfreezeUser`, `GrantBadge`

- **Локации:**
  - `backend/internal/delivery/http/admin/handler.go:133-158` (`FreezeUser`, `UnfreezeUser`)
  - `backend/internal/delivery/http/admin/handler.go:180-197` (`GrantBadge`)

- **Что:** В `CLAUDE.md:272` явно: *«TOTP 2FA обязательна для всех destructive actions, 12h TTL по дефолту.»* Реально TOTP проверяется только в `ResetPassword`, `ChangeTier`, `RevokeBadge`, `DeleteFeedback`, `UpdateFeedbackStatus`. `FreezeUser` / `UnfreezeUser` / `GrantBadge` — **нет вызова `h.verifyTOTP(r)`**. Frontend это знает (`admin/users.tsx` не показывает TOTP-prompt для freeze).

- **Последствия (exploit с украденным admin JWT):**
  1. Атакующий получает admin JWT (XSS или украденный токен).
  2. JWT валиден 15 минут (см. `cfg.JWT.access_duration`).
  3. Без 2FA-подтверждения атакующий может:
     - `POST /api/admin/users/{any_user_id}/freeze` — заморозить любого юзера, **в том числе других админов** (см. CR-4 ниже).
     - `POST /api/admin/users/{self_id}/badges/{badge_id}/grant` — выдать «бейдж администратора» (если такой есть в catalog'е) → escalation видимости.

- **CVSS 3.1:** **8.1** (AV:N / AC:L / PR:H / UI:N / S:U / C:H / I:H / A:H — атакующий уже имеет admin JWT, но 2FA должна быть defence-in-depth).

- **Best practice:** [OWASP Authentication Cheat Sheet — MFA](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html#multi-factor-authentication). Цитата: *"MFA bypass conditions must be explicit and audited."*

- **Fix:**
  ```go
  // backend/internal/delivery/http/admin/handler.go:135 (FreezeUser)
  func (h *Handler) FreezeUser(w http.ResponseWriter, r *http.Request) {
      if err := h.verifyTOTP(r); err != nil {
          httperr.RespondWithRequest(w, r, err)
          return
      }
      // ... existing
  }
  // Аналогично UnfreezeUser:151, GrantBadge:182
  ```

  И обновить request-DTO (добавить `TOTPCode string `validate:"required,len=6"``), и frontend `admin/users.tsx` показывать TOTP-prompt.

- **Effort:** **S** (3 handler'а + 3 DTO + 1 frontend dialog) ≈ 1 час.

### CR-4 [SECURITY] — Admin может заморозить другого admin'а (нет защиты от cross-admin freeze)

- **Локация:** `backend/internal/usecases/admin/admin.go:117-151` (`FreezeUser`)

- **Что:** Защита `targetID == info.AdminID` (anti-self-freeze) есть, но **нет** проверки `target.Role == admin`. Один admin может заморозить остальных админов и стать единственным контролирующим систему.

- **PoC (insider threat):** Compromised/disgruntled admin Alice → `POST /api/admin/users/{bob_admin_id}/freeze` → Bob выкидывается. Alice одна управляет всей системой и платежами.

- **CVSS:** 5.7 (insider, audit log фиксирует — но Bob не сможет залогиниться, чтобы это увидеть, пока Alice не разблокирует).

- **Fix:**
  ```go
  // usecases/admin/admin.go:128 (после Self-freeze check)
  target, err := s.users.GetByID(ctx, targetID)
  if err != nil { return err }
  if target.IsAdmin() {
      return ErrCannotFreezeAdmin
  }
  ```

  Если нужны cross-admin freezes (например, для off-boarding) — ввести super-admin role или dual-control (требовать одобрения двух admin'ов).

- **Effort:** **S** (1 проверка + 1 sentinel error).

### CR-5 [SECURITY] — TOCTOU race-condition обхода квот при concurrent создании ресурсов

- **Локации:**
  - `backend/internal/usecases/prompt/prompt.go:45-95` (`CheckPromptQuota` → `prompts.Create`)
  - `backend/internal/usecases/collection/collection.go:35-50` (CheckCollectionQuota → Create)
  - `backend/internal/usecases/chain/chain.go:60-75` (`CheckChainQuota` → `chains.Create`)
  - `backend/internal/usecases/quota/quota.go:55-194` — все `Check*` реализованы через `SELECT COUNT(*)` без блокировки.

- **PoC (Free plan, `MaxPrompts=10`, у юзера 9 промптов):**
  ```bash
  echo {1..10} | xargs -n1 -P10 -I{} curl -X POST -H "Authorization: Bearer $JWT" \
       -d '{"title":"P{}","content":"x"}' /api/prompts
  ```
  Все 10 запросов одновременно вызывают `CheckPromptQuota` → `quotaRepo.CountPrompts` возвращает 9 → `9 < 10`, все 10 проходят check → 10 INSERT'ов идут в БД → у юзера 19 промптов вместо 10.

- **CVSS 3.1:** **7.1** (Integrity High, без auth-escalation, требует параллельные req — но xargs тривиален).

- **Best practice:** [PostgreSQL `SELECT ... FOR UPDATE`](https://www.postgresql.org/docs/current/sql-select.html#SQL-FOR-UPDATE-SHARE) или advisory locks. Кому очень не хочется advisory — partial UNIQUE index с pre-allocated row counter.

- **Fix (минимально-инвазивный):**
  ```go
  // backend/internal/usecases/prompt/prompt.go:45 (Create)
  return s.prompts.InTransaction(ctx, func(txRepo PromptRepository) error {
      // SELECT FOR UPDATE на user-row или advisory lock
      if err := txRepo.AcquireUserAdvisoryLock(ctx, in.UserID); err != nil {
          return err
      }
      used, err := txRepo.CountByUser(ctx, in.UserID)  // в той же tx
      if err != nil { return err }
      if used >= int64(plan.MaxPrompts) {
          return quota.ErrPromptsExceeded
      }
      return txRepo.Create(ctx, p)
  })
  ```
  Где `AcquireUserAdvisoryLock`:
  ```go
  func (r *promptRepo) AcquireUserAdvisoryLock(ctx context.Context, userID uint) error {
      return r.db.WithContext(ctx).Exec("SELECT pg_advisory_xact_lock(?, ?)", lockNamespace, int64(userID)).Error
  }
  ```
  Advisory lock освобождается на `COMMIT/ROLLBACK` транзакции — race window закрывается.

- **Effort:** **M** (1 helper + миграция вызовов в Create методах prompt/collection/chain) ≈ 3-4 часа.

### CR-6 [SECURITY] — Email normalization отсутствует — duplicate accounts + OAuth-link mismatch

### CR-7 [RELIABILITY/PERF] — Отсутствует общий `pkg/background.Pool` для fire-and-forget горутин

- **Локации (все запускают raw `go func()` без WaitGroup и без подписки на `App.Shutdown`):**
  - `backend/internal/usecases/auth/oauth.go:308-314` (`touchLastLogin` после OAuth callback)
  - `backend/internal/usecases/team/team.go:217-221` (отправка invite-email)
  - `backend/internal/usecases/apikey/apikey.go:154-165` (`UpdateLastUsed`)
  - `backend/internal/usecases/share/share.go:238-249, 254-256, 290-311` (view counter + share_view_log если Pro+)

- **Что:** На burst-нагрузке (виральная share-ссылка, массовый OAuth login после deploy) количество goroutine'ов не ограничено. На VPS 4GB при 1000+ goroutine ~8MB на каждую = 8MB+ overhead, плюс connection pool пожирается (см. CR-8). На graceful shutdown все goroutines прерываются posix-signal'ом — `users.TouchLastLogin` может прервать TCP в середине UPDATE.

- **Последствия:**
  - На share-burst: лавина concurrent connections к PG (см. CR-8).
  - На SIGTERM: orphan-операции (logo upload `team.go:217` потеряется, view-counter не запишется).
  - 0 наблюдаемости (нет metrics на background queue depth).

- **Best practice:** [Russ Cox — «Never start a goroutine without knowing how it will stop»](https://research.swtch.com/gomutex). Worker-pool с buffered channel + WaitGroup.

- **Fix:**
  ```go
  // backend/internal/pkg/background/pool.go
  type Pool struct {
      jobs   chan func(context.Context)
      wg     sync.WaitGroup
      ctx    context.Context
      cancel context.CancelFunc
  }
  func New(workers int, queue int) *Pool {
      ctx, cancel := context.WithCancel(context.Background())
      p := &Pool{
          jobs:   make(chan func(context.Context), queue),
          ctx:    ctx,
          cancel: cancel,
      }
      for range workers {
          p.wg.Add(1)
          go func() {
              defer p.wg.Done()
              for job := range p.jobs {
                  safeloop.RunWithRecover("background-pool", func() { job(p.ctx) })
              }
          }()
      }
      return p
  }
  func (p *Pool) Submit(job func(context.Context)) {
      select {
      case p.jobs <- job:
      default:
          metrics.BackgroundDrops.Inc() // best-effort, не блокируем основной запрос
      }
  }
  func (p *Pool) Shutdown(timeout time.Duration) error {
      close(p.jobs)
      done := make(chan struct{})
      go func() { p.wg.Wait(); close(done) }()
      select {
      case <-done:
          p.cancel()
          return nil
      case <-time.After(timeout):
          p.cancel()
          return errors.New("background pool shutdown timeout")
      }
  }
  ```

  В `app.App`: `bgPool := background.New(16, 256)`, инжектится в `share`/`apikey`/`team`/`auth.OAuth`. В `App.Shutdown(ctx)` вызывается `bgPool.Shutdown(30 * time.Second)`. Все `go func() {...}()` заменяются на `s.bg.Submit(func(ctx context.Context) {...})`.

- **Effort:** **M** (~2 часа: helper + замены в 5 местах + тест graceful shutdown).

### CR-9 [PERF] — N+1 в `chain.StartExecution` (ужe preload'енные prompts перезагружаются)

- **Локация:** `backend/internal/usecases/chain/chain.go:821-837`

- **Что:**
  ```go
  // chain.go:821
  for _, step := range c.Steps {
      if step.PromptID == nil { continue }
  ```

  ```go
      if step.PromptID == nil { continue }
      if _, ok := promptContents[*step.PromptID]; ok { continue }
      // step.Prompt уже preload'ен через Preload("Steps.Prompt")
      if step.Prompt.ID == 0 {
          return nil, ErrPromptNotFound  // soft-deleted между Get и Start
      }
      promptContents[*step.PromptID] = step.Prompt.Content
  }
  ```

### CR-10 [PERF] — `analyticsRepo.PossibleDuplicates` потенциально O(N²) под ежесуточным batch-burst

- **Локация:** `backend/internal/infrastructure/postgres/repository/analytics_repo.go:415-446`

- **Что:** Запрос `JOIN prompts a ON prompts b ON a.id < b.id` — перебор всех пар. Защита `a.updated_at >= NOW() - INTERVAL '30 days'`, но **только для `a`**.   -- analytics_repo.go:415
  ```sql
  AND a.updated_at >= NOW() - INTERVAL '30 days'
  AND b.updated_at >= NOW() - INTERVAL '90 days'  -- было: только a
  AND a.content %% b.content                       -- использует GIN index
  AND similarity(a.content, b.content) >= ?
  ```
  И `statement_timeout` per-юзер:
  ```go
  ctx, cancel := context.WithTimeout(parent, 30*time.Second)
  defer cancel()
  // ... PossibleDuplicates(ctx, ...)
  ```

### MJ-17 [MIGRATION] — `000051_prompts_fts.up.sql` — `ADD COLUMN GENERATED ALWAYS AS ... STORED` блокирует таблицу под AccessExclusiveLock

- **Локация:** `backend/internal/infrastructure/postgres/migrations/000051_prompts_fts.up.sql`

- **Fix:**
  1. Миграция 000062: `ADD COLUMN search_tsv tsvector` (без GENERATED) — быстро, lock метаданных.
  2. Backfill в коде (Go) chunks по 1000 rows: `UPDATE prompts SET search_tsv = to_tsvector(...) WHERE id IN (...)`.
  3. Миграция 000063: переключить на GENERATED ALWAYS через `ALTER COLUMN ... SET GENERATED ALWAYS AS ...`.
  4. Миграция 000064: `CREATE INDEX CONCURRENTLY` на `search_tsv` (требует `-- migrate:no-transaction`).

  Альтернатива: смириться с downtime и зафиксировать в runbook + запускать в low-traffic window.

- **Effort:** **L** (~1 день: 3 миграции + backfill код + runbook).

## Architectural Observations (long-term)

1. **Composition root (`app/app.go`)** сделан грамотно. Все зависимости явно собираются, late-binding через `Set*` методы (SetActivity, SetEmail, SetTierChangeNotifier, SetFeedbackRepository, SetLogoRepo). Pattern «optional setter» правильно используется для разрыва циклических зависимостей. Composition-root правильно владеет `*gorm.DB`, ниже его никто не видит — за исключением CR-1/MJ-1 и MJ-2.

2. **Models — чистый пакет без циркулярок**. Импортируют только `time` и `gorm`. 27 файлов, никаких circular deps.

3. **Repository интерфейсы (interface/repository/)** — чистые. 26 файлов, gorm-импорта нет. `errors.go` правильно вводит `repo.ErrNotFound`, `repo.ErrConflict`. Из 23 реализаций корректно мапят 21; 2 файла утечкают gorm (см. MJ-1).

4. **Local interface pattern в `delivery/http/admin/handler.go:38-69`** (`AdminService`, `TOTPVerifier`, `AuditReader`, `HealthCounter`) — отличный пример Go idiomatic dependency-inversion. Этот паттерн стоит распространить на остальные handlers (сейчас часть берёт concrete `*Service`).

5. **Late-binding через setters** — компромисс для разрыва circular DI. Альтернатива — Functional Options Pattern (`NewService(repo, ..., WithActivity(svc), WithEmail(email))`).

6. **Usecase auth/auth.go (664 строк, 26 методов)** — пограничный кандидат на разбиение. Можно вынести `password.go` (Set/Forgot/Reset/Change).

7. **CRON_FAST_DEV паттерн в `app.go:233-243`** — элегантное решение для QA. Все loops ускоряются с часов до секунд. Учитывает `cfg.Server.IsDev()` гейтинг.

8. **Use of `panic` в `app.go:177, 408, 417` для fail-fast при загрузке embedded catalog'ов** — правильный подход.

9. **CLAUDE.md**: 30% backend-фич устарели (CR-1 doc-drift, см. секцию «Расхождения с CLAUDE.md»).

10. **`mcpserver` исключён из ревью** (по запросу). Использует go-sdk v1.5 через 4 интерфейса — правильная декаплинг-граница.

11. **Frontend**: основные patterns правильные (Suspense + lazy для тяжёлых страниц, manualChunks, centralized fetch wrapper, optimistic updates). Главные системные проблемы:
    - Отсутствие React Compiler (`babel-plugin-react-compiler`) → ручной useCallback/useMemo обязателен для perf-чувствительных мест.
    - ESLint без `eslint-plugin-jsx-a11y` → a11y-issues растут.
    - Backend ошибки приходят как `{"error": "msg"}` без `field` — невозможен field-level mapping в формах. Cross-stack изменение нужно для лучшего UX.

12. **Migrations**: 61 миграция применена, паттерны идиоматичные. Главная зона риска — большие `prompts`, `prompt_usage_log` таблицы при будущих index-additions без CONCURRENTLY.

13. **CI/CD**: pipeline работает, есть rollback в deploy. Главные проблемы — supply-chain (third-party actions, govulncheck декоративный, npx @latest), отсутствие integration-tests в CI.
