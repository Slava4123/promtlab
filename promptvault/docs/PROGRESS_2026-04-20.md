## 1. Что задеплоено на prod (v1.3.6, promtlabs.ru)

### Backend OAuth 2.1 — полностью рабочий

**Endpoints (смонтированы в `app.go`, проксируются nginx):**

| Путь | RFC | Описание |
|------|-----|----------|
| `POST /oauth/register` | RFC 7591 | Dynamic Client Registration |
| `GET /oauth/authorize` | OAuth 2.1 | PKCE S256 authorize, читает `refresh_token` cookie или редиректит на `/sign-in?return_url=...` |
| `POST /oauth/token` | OAuth 2.1 | Exchange code / refresh |
| `POST /oauth/revoke` | RFC 7009 | Token revocation |
| `GET /.well-known/oauth-protected-resource` | RFC 9728 | Discovery metadata |
| `GET /.well-known/oauth-authorization-server` | RFC 8414 | AS metadata |

`/mcp` endpoint отдаёт `WWW-Authenticate` header per RFC 9728 §5.1 —
Claude.ai автоматически найдёт OAuth flow через discovery chain.

**Файлы (всё в `promptvault/backend/`):**
- `internal/models/oauth.go` — 3 GORM-сущности (OAuthClient, OAuthAuthorizationCode, OAuthToken)
- `internal/models/policy.go` — shared scope-тип (ReadOnly, TeamID, AllowedTools)
- `internal/interface/repository/oauth.go` — repo-интерфейсы
- `internal/infrastructure/postgres/repository/oauth_repo.go` — GORM impl
- `internal/infrastructure/postgres/migrations/000037_oauth_server.{up,down}.sql`
- `internal/pkg/pkce/pkce.go` — PKCE S256 verify (RFC 7636)
- `internal/pkg/tokens/tokens.go` — opaque tokens (pvoat_/pvort_/pvoac_/pvoci_)
- `internal/usecases/oauth_server/` — бизнес-логика
- `internal/delivery/http/oauth_server/handler.go` — HTTP-эндпоинты
- `internal/delivery/http/metadata/handler.go` — `.well-known/*`
- `internal/mcpserver/auth.go` — dual-path (pvlt_ API keys OR pvoat_ OAuth access)

## 2. Незакомиченные изменения (блокируют Claude.ai тест)

### Файлы в working tree, не в origin

```
M promptvault/frontend/src/pages/sign-in.tsx
M promptvault/frontend/src/pages/oauth-callback.tsx
```

### Что меняют

- `sign-in.tsx`: при клике на «Войти через GitHub/Google/Yandex»
  сохраняет текущий `?return_url=...` в `sessionStorage["pv:oauth_return_url"]`
  перед redirect на OAuth provider.
- `oauth-callback.tsx`: после успешного OAuth login читает
  `sessionStorage["pv:oauth_return_url"]` и делает `window.location.href`
  туда вместо `navigate("/dashboard")`. Одноразовое использование (removeItem).

### Зачем

Когда Claude.ai открывает наш `/oauth/authorize`, мы редиректим на
`/sign-in?return_url=/oauth/authorize?...`. Если пользователь логинится
через Google, OAuth-client flow делает redirect на Google → callback на
`/oauth/callback`, теряя `return_url` из query. Без фикса пользователь
после Google-логина оказывается на `/dashboard`, а не возвращается к
authorize flow для Claude.

### Что делать завтра

1. **`git status`** — должны быть два изменённых файла.
2. **`cd promptvault/frontend && npx tsc -b --noEmit`** — убедиться что TS без ошибок (локально проверено).
3. Bump `promptvault/server.json`: `"1.3.6"` → `"1.3.7"`.
4. Bump `promptvault/backend/internal/mcpserver/server.go`: `Version: "v1.3.7"`.
5. Коммит:
   ```
   git add promptvault/frontend/src/pages/sign-in.tsx \
           promptvault/frontend/src/pages/oauth-callback.tsx \
           promptvault/server.json \
           promptvault/backend/internal/mcpserver/server.go
   git commit -m "fix(frontend): OAuth client-flow сохраняет return_url через sessionStorage"
   ```
6. `git push origin main && git tag v1.3.7 && git push origin v1.3.7`
7. Дождаться deploy (CI/CD workflow) ~5 мин.

## 3. Тест Claude.ai custom connector (после деплоя v1.3.7)

### Шаги

1. **Hard reload** любой открытой `promtlabs.ru` вкладки (`Ctrl+Shift+R`).
2. Открыть https://claude.ai/settings/connectors
3. В карточке **ПромтЛаб (CUSTOM)** нажать **Connect**
4. Откроется popup → будут варианты:
   - **Если уже залогинены в promtlabs.ru** (cookie есть): сразу issue code → Claude закроет popup → 30 tools появятся.
   - **Если нет сессии**: редирект на `/sign-in?return_url=...` → залогиниться (email/password или Google/GitHub/Yandex) → автоматический возврат на authorize → Claude получит code → tools.
5. Проверить в Claude — спросить «покажи мои промпты» — должен вызвать `list_prompts` и вернуть содержимое.

### Если что-то не работает

- Проверить: `curl -i "https://promtlabs.ru/oauth/authorize?response_type=code&client_id=x&redirect_uri=https://claude.ai/api/mcp/auth_callback&code_challenge=abc&code_challenge_method=S256"` → должен вернуть **302** на `/sign-in?return_url=...`.
- Проверить: `curl -i "https://promtlabs.ru/oauth/callback"` → должен вернуть **200 HTML** (SPA), НЕ 404.
- Проверить Network tab в Claude.ai popup — что именно возвращается и на каком шаге.

### Potential issue до подачи: нет consent screen

При OAuth authorize сейчас: cookie → сразу code. Это CSRF-риск
(вредоносный сайт top-level redirect → authorize от имени залогиненного
user'а → code для чужого client_id).

**Рекомендуется добавить consent screen:**
- Новая SPA-страница `/oauth/consent?client_id=...&scope=...&state=...`
- Backend `/oauth/authorize` редиректит ТУДА (а не сразу issue code)
- Пользователь видит «Разрешить *Claude* доступ к вашим промптам? [Allow] [Deny]»
- На Allow → POST `/oauth/authorize/consent` → backend issues code → redirect к Claude

Оценка: ~2-4 часа работы (backend + SPA page + UX).

## 7. Known issues (в коде, задокументированы, не блокеры для MVP)

### Security
- **Нет consent screen на `/oauth/authorize`** — CSRF-риск. Фикс: новая SPA-page + backend flow разделение. См. §4.
- **client_secret не валидируется** для `auth_method=client_secret_post`. На практике DCR создаёт `auth_method=none`, так что defense-in-depth.

### Spec deviations (работают, но не 100% RFC)
- **Errors в JSON** вместо redirect per RFC 6749 §4.1.2.1. Проще в debug, менее совместимо.
- **`response_type` проверяется поздно** — после cookie+client lookup. Не баг, perf-мелочь.

### UX
- **Frontend cache** — пользователи могут видеть старый bundle после деплоя до hard-reload. Решение: добавить `Cache-Control: no-cache` на `index.html` в nginx. Vite уже делает hash в filename для ассетов, но `index.html` должен обновляться сразу.
